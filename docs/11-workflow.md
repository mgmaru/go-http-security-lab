# 開発ワークフロー

各Phaseでは、次の順序で進めます。

### 1. 要求と制約を整理する

実装前に、次を自分の言葉で記載します。

```text
何を実現するか
誰が利用するか
どのような入力があるか
どのような結果を返すか
何を今回の対象外にするか
守る必要がある制約は何か
```

要件に書かれていない挙動を勝手に決めず、不明点は「自分で決める仕様」として明示します。

### 2. ユースケースと失敗ケースを書く

正常な利用だけでなく、失敗や悪用も先に考えます。

```text
正常に完了する流れ
入力が不正な場合
対象が存在しない場合
未ログインの場合
権限がない場合
処理途中で失敗した場合
同じ処理が再送された場合
```

### 3. 設計案を複数考える

重要な判断について、少なくとも2案を比較します。

比較観点の例：

```text
単純さ
変更しやすさ
テストしやすさ
依存関係
データ整合性
セキュリティ
実装コスト
現在のPhaseに対して過剰ではないか
```

ここでは、具体的な完成コードを書かず、責務・境界・データの流れを考えます。

### 4. 採用案を決めて記録する

どの案を採用するかを決め、理由と欠点を設計メモへ残します。

重要な判断はADRへ記録します。

```text
なぜ採用したか
何を犠牲にしたか
どの条件になったら見直すか
```

### 5. API契約と受け入れ条件を決める

実装前に、外部から観察できる挙動を決めます。

```text
HTTPメソッド
URL
必要なヘッダー
入力形式
成功時のステータスコード
失敗時のステータスコード
レスポンス形式
認証・認可条件
```

あわせて、Phaseの完了条件をテスト可能な表現へ落とします。

### 6. 最小の正常系を実装する

まず、そのPhaseの中心となる一つのユースケースを動かします。

この段階では、将来の拡張を先回りして大きな抽象化を作りません。

### 7. 異常系と境界条件を実装する

各APIについて、最低でも5個以上の異常系を考えます。

例：

```text
空のボディ
壊れたJSON
必須項目なし
長すぎる入力
未知のフィールド
未ログイン
権限不足
存在しないID
不正なContent-Type
未対応のHTTPメソッド
```

### 8. curlでHTTP契約を確認する

送信したリクエストと受信したレスポンスを、設計したAPI契約と比較します。

想定と異なった場合は、コードだけでなく設計メモも見直します。

### 9. APIクライアントへ保存する

Phase 2以降では、`curl`で内容を理解したリクエストをBrunoへ保存します。

```text
正常系
主要な異常系
未ログイン
権限不足
ユーザーA・ユーザーB・管理者
```

Brunoはリクエストを整理・再利用するために使い、HTTPの理解や自動テストの代わりにはしません。

環境ファイルへ本物のパスワード、セッションID、APIキーを保存しないことを確認します。

### 10. httptestを書く

正常系だけでなく、失敗時とセキュリティ上の境界をテストします。

テストは、設計上守りたい契約を実行可能な形で表現するものとして扱います。

Brunoに保存した重要な手動確認は、可能な範囲でGoの自動テストへ変換します。

### 11. Burp Suiteで設計の前提を壊す

ブラウザや`curl`が通常は送らない値も含めて、通信を改変します。

```text
IDの差し替え
Cookieの削除・変更
HTTPメソッドの変更
未知のJSON項目
権限値の書き換え
同じリクエストの再送
```

ここでは「クライアントは正しい値を送る」という設計上の思い込みを検証します。

### 12. 責務と依存関係を振り返る

機能追加後に、次を確認します。

```text
一つの関数や型が多くの責務を持っていないか
HTTP固有処理と業務ルールが混ざっていないか
データ保存方式が上位の処理へ漏れていないか
テストが書きにくい依存関係になっていないか
変更理由が異なる処理を同じ場所へ置いていないか
```

問題が実際に現れた箇所だけをリファクタリングします。

### 13. 設計判断と学習内容を更新する

実装前の予想と、実装後に分かったことを比較します。

```text
採用した設計は実装しやすかったか
異常系を追加しやすかったか
テストしやすかったか
セキュリティ上の抜けがなかったか
変更しにくかった箇所はどこか
次回は何を変えるか
```

設計変更があった場合は、設計メモまたはADRを更新します。

### 14. AIへレビューを依頼する

自分なりの設計、実装、テスト、手動確認が一通り終わった後で、必要に応じてAIへレビューを依頼します。

相談前に、次を自分の言葉で記録します。

```text
現在の設計
採用した理由
比較した選択肢
自分で試したこと
納得できている点
疑問に感じている点
AIに確認したい観点
```

AIには、最初から完成コードを求めず、問題点、反対意見、見落とし、過剰設計の可能性を尋ねます。

回答を受けた後は、採用した内容と採用しなかった内容を理由とともに`docs/ai/consultations/`へ記録します。

### 15. Phaseの振り返りを作成する

作業中のメモを整理し、`docs/retrospectives/`へPhaseごとの振り返りを作成します。

最低限、次を記録します。

```text
何を実装したか
何が起きたか
何を学んだか
何が分からなかったか
何に迷ったか
設計から何を変更したか
次へ持ち越すこと
```

未解決の疑問は`docs/questions/unresolved-questions.md`へ登録します。

将来にも影響する設計判断は、必要に応じてADRへ移します。

### 16. 期間と進捗を更新する

Phaseの開始日、完了日、作業時間、予定との差を`docs/progress/`へ記録します。

予定との差が大きい場合は、単に遅れとして扱わず、次を振り返ります。

```text
要求の理解に時間がかかったか
設計変更が多かったか
テストケースが想定より多かったか
調査へ時間を使ったか
過剰なリファクタリングをしていないか
利用できる学習時間が変化したか
```

この結果を、次のPhaseの見積もりへ反映します。

## 7. Gitブランチの例

フェーズごとにブランチを作成します。

```text
feature/phase01-basic-http
feature/phase02-task-crud
feature/phase03-middleware
feature/phase04-database
feature/phase05-authentication
feature/phase06-authorization
feature/phase07-admin
feature/phase08-file-upload
feature/phase09-burp-testing
feature/phase10-security-tests
feature/phase11-frontend
feature/phase12-browser-security
feature/phase13-concurrency
feature/phase14-ci
```

## PRに記載する内容

```text
解決しようとした問題
要求・制約・対象外
検討した設計案
採用した設計と理由
追加した外部ライブラリと採用理由
責務分割と依存関係
何を実装したか
どのHTTP仕様を学んだか
正常系と異常系
セキュリティ上の信用境界
追加したテスト
curlによる確認方法
追加・更新したBrunoリクエスト
Brunoから自動テストへ移した項目
Burp Suiteで試したこと
実装後に見直した設計判断
AIへ相談する前の自分の考え
AIから得た指摘
採用した提案・採用しなかった提案と理由
今回学んだこと
分からなかったこと・迷ったこと
更新したRetrospective・ADR・未解決事項
残っている課題
```

実際に一人で進める場合でも、PR形式の説明を残すことで、転職活動時に設計意図や学習内容を説明しやすくなります。

## 8. コミット方針

一つのコミットに多くの変更を詰め込みすぎないようにします。

例：

```text
feat: add health endpoint
test: add health handler tests
feat: add JSON response helper
fix: reject multiple JSON values
feat: add task creation endpoint
test: add invalid task payload cases
security: prevent access to other users tasks
docs: add Burp authorization test notes
```
