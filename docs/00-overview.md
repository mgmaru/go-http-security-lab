## 最終的に作るアプリ

複数ユーザーに対応したタスク管理Web APIを作ります。

### 主な機能

- ユーザー登録
- ログイン
- ログアウト
- セッション管理
- タスクの作成・取得・更新・削除
- ユーザーごとのデータ分離
- 管理者機能
- プロフィール画像アップロード
- 操作ログ
- 同時更新制御
- セキュリティテスト
- 最小限のWebフロントエンド

### 想定API

```http
GET    /health
GET    /hello
POST   /echo

POST   /api/users
POST   /api/login
POST   /api/logout
GET    /api/me

GET    /api/tasks
GET    /api/tasks/{id}
POST   /api/tasks
PUT    /api/tasks/{id}
PATCH  /api/tasks/{id}
DELETE /api/tasks/{id}

GET    /api/admin/users
GET    /api/admin/tasks
PATCH  /api/admin/users/{id}/role
DELETE /api/admin/users/{id}

POST   /api/profile/image
```

## 最終ゴール

このプロジェクトのゴールは、単にAPIを完成させることではありません。

次の内容を、自分の言葉とコードで説明できる状態を目指します。

- 要求、制約、対象外を整理する方法
- 複数の設計案を比較する方法
- 責務と依存関係を決める方法
- 設計判断とトレードオフを記録する方法
- 実装とテストから設計を改善する方法
- 学習中の出来事・失敗・迷いを振り返る方法
- 未解決の疑問を管理し、後から解決へつなげる方法
- 自分の理解と判断の変化を説明する方法
- AIを思考の代替ではなくレビューと批評に使う方法
- AIの提案を評価し、最終判断を自分の言葉で説明する方法
- 作業期間を見積もり、実績との差から計画を改善する方法
- HTTPリクエストとレスポンスの構造
- メソッドとステータスコードの使い分け
- APIクライアントでリクエストを整理・再利用する方法
- 手動確認、APIコレクション、自動テスト、セキュリティ検査を使い分ける方法
- Cookieとセッションの関係
- 認証と認可の違い
- フロントエンドの入力値を信用できない理由
- 他人のデータを操作できないようにする方法
- SQLインジェクションを防ぐ方法
- ファイルアップロードの危険性
- CORSとCSRFの違い
- XSSが発生する理由
- Burp Suiteで通信を検査する方法
- 手動で見つけた問題を自動テストへ変換する方法
- セキュリティを後付けではなく、設計・実装・テストへ組み込む方法

最終的には、次の学習サイクルを自然に回せるようにします。

```text
考える
  ↓
設計する
  ↓
作る
  ↓
見る
  ↓
壊す
  ↓
直す
  ↓
テストする
  ↓
自分の疑問を言語化する
  ↓
AIからレビューを受ける
  ↓
自分で最終判断する
  ↓
振り返る
  ↓
未解決事項を残す
  ↓
判断と学びを説明する
```
