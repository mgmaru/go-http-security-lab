# テスト戦略

このプロジェクトでは、テスト数を増やすことではなく、設計上守りたい契約を適切な場所で検証することを目的とします。

## テストと検証手段の役割

| 種類 | 主な対象 | 目的 |
|---|---|---|
| Unit Test | バリデーション、変換、業務ルール | 小さな規則を高速に確認する |
| Handler Test | HTTPハンドラー | メソッド、ヘッダー、入力、ステータス、JSONを確認する |
| Middleware Test | 横断処理 | ログ、Recovery、認証、リクエストIDを確認する |
| Repository Test | DBアクセス | SQL、制約、トランザクション、ロールバックを確認する |
| Integration Test | APIからDBまで | 複数層の連携を確認する |
| Security Test | 認証・認可・入力改変 | 攻撃ケースの再発を防ぐ |
| `curl` | HTTPの手動確認 | リクエストを明示して仕様を理解する |
| Bruno | 手動テストケース集 | リクエストを保存・整理・再利用する |
| Burp Suite | 改変・再送 | 通常のクライアントが送らない値を試す |
| Race Detector | 並行処理 | データ競合を検出する |

## 基本原則

- 同じ挙動をすべての層で重複してテストしない
- 重要な業務ルールは、可能な限り高速なテストで確認する
- HTTP契約はHandler TestまたはIntegration Testで確認する
- DB固有の制約とトランザクションはRepository Testで確認する
- 手動で見つけた重要な問題は自動テストへ移す
- 正常系だけでなく、失敗時と境界条件を確認する
- テストが実行順序や共有状態に依存しないようにする
- テストが通ることを安全性の完全な証明とは考えない

## テストケースを考える順序

```text
要求から正常系を決める
  ↓
入力の境界を考える
  ↓
存在しない対象を考える
  ↓
未認証・権限不足を考える
  ↓
途中失敗と再送を考える
  ↓
Burp Suiteで想定外の改変を試す
  ↓
重要な発見を自動テストへ移す
```

## 最低限確認する観点

- 正常な入力
- 必須値の欠落
- 空文字、ゼロ値、境界値
- 長すぎる入力
- 壊れたJSON
- 未知のフィールド
- 不正な`Content-Type`
- 未対応のHTTPメソッド
- 存在しないID
- 未ログイン
- 他人のリソース
- 管理者権限の不足
- 同じリクエストの再送
- 処理途中の失敗
- 同時更新
- ログやレスポンスへの秘密情報漏えい

## テストを書く前の問い

- このテストは、どの要求または設計判断を守るのか
- より小さく高速なテストで確認できないか
- 実装詳細ではなく、観察可能な振る舞いを確認しているか
- テストデータの準備が難しい原因は設計にないか
- この問題が再発したとき、本当に検出できるか

## Phase完了時の確認

- [ ] 中心的な正常系が自動テストされている
- [ ] 主要な異常系と権限境界がテストされている
- [ ] 手動検査で見つけた重要な問題を自動化した
- [ ] テストが実行順序に依存していない
- [ ] DBやグローバル状態がテスト間で分離されている
- [ ] `go test ./...`が成功する
- [ ] 必要なPhaseでは`go test -race ./...`が成功する
