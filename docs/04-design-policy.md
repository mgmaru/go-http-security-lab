# 設計方針

### 2.4 設計も実装と同じく反復して学ぶ

設計力は、最初から正解の構造を当てる能力ではありません。

このプロジェクトでは、設計を次のように扱います。

```text
要求から設計上の問題を見つける
  ↓
少なくとも2つの案を考える
  ↓
制約とトレードオフを比較する
  ↓
現時点の採用案を決める
  ↓
実装とテストで妥当性を確かめる
  ↓
必要であれば設計を変更する
```

設計資料には、完成したコードの実装手順やアルゴリズムを先に書きません。

記録するのは、主に次の内容です。

- 何を実現する必要があるか
- 何を今回の対象外にするか
- どの責務をどこへ置くか
- どのデータを誰が所有するか
- どのような失敗が起こり得るか
- どの境界を信用しないか
- どの設計案を比較したか
- なぜ現在の案を採用したか
- どのテストで設計の妥当性を確認するか

これにより、実装方法そのものは自分で考えながら、設計判断を言語化する練習ができます。

### 設計課題の進め方

各Phaseでは、実装前に短い設計メモを作成します。長大な設計書ではなく、実装前後で判断を比較できる程度の内容にします。

#### 設計メモに記載する内容

```text
目的
今回の対象範囲
対象外にすること
ユースケース
APIやデータの契約
主な責務と境界
依存関係
想定する失敗
セキュリティ上の信用境界
検討した設計案
採用案と理由
欠点・トレードオフ
テスト方針
未解決事項
```

すべての項目を毎回詳しく書く必要はありません。そのPhaseで重要な項目を選んで記載します。

#### ADRを残す判断

後から変更理由を知りたくなる重要な判断は、Architecture Decision Recordとして残します。

例：

- セッション方式をどのようにするか
- 認可をどの層で行うか
- Repository層を導入するか
- `403`と`404`をどう使い分けるか
- ファイルをどこへ保存するか
- 同時更新をどの方式で制御するか

ADRには、少なくとも次を記載します。

```text
背景
決める必要があること
検討した選択肢
採用した選択肢
採用理由
想定される影響
後から見直す条件
```

### Phaseごとの主な設計テーマ

| Phase | 設計テーマ |
|---|---|
| Phase 0 | クライアントとサーバーの境界、HTTPメッセージの責務 |
| Phase 1 | API契約、エラー形式、共通処理と個別処理の境界 |
| Phase 2 | リソース設計、URL設計、`PUT`と`PATCH`、バリデーション方針 |
| Phase 3 | 横断的関心事、ミドルウェアの責務と実行順序 |
| Phase 4 | 永続化境界、Repositoryの責務、トランザクション境界 |
| Phase 5 | 認証フロー、セッションのライフサイクル、秘密情報の管理 |
| Phase 6 | 認可ポリシー、データ所有権、存在情報を隠すかどうか |
| Phase 7 | ロール設計、管理者権限の不変条件、監査ログ |
| Phase 8 | ファイルの信用境界、保存責務、検証と配信の分離 |
| Phase 9 | 脅威モデル、攻撃経路、手動検査の優先順位 |
| Phase 10 | テスト階層、テストデータ生成、再現性と独立性 |
| Phase 11 | フロントエンドとAPIの責務分担、画面状態と通信エラー |
| Phase 12 | オリジン境界、ブラウザの信頼モデル、CSRF・XSS対策 |
| Phase 13 | 整合性、競合制御、タイムアウト、キャンセル伝播 |
| Phase 14 | 実行環境、設定管理、可観測性、デプロイと運用 |

### Goソースコードの配置方針

このプロジェクトでは、リポジトリ直下に`go.mod`を置き、単一のGoモジュールとして進めます。

特別な理由がない限り、アプリのソースコードをまとめるためだけの`src/`は作りません。

```text
go-http-security-lab/
├── go.mod
├── cmd/
├── internal/
├── docs/
└── README.md
```

`src/`を使わない理由は、Goでは`go.mod`が置かれた場所を基準にパッケージが構成されるためです。

リポジトリ直下に`go.mod`がある状態で`src/`を追加すると、`src`がパッケージパスの一部になります。

```text
github.com/example/go-http-security-lab/src/...
```

今回の単一Goアプリでは、この階層を追加する明確な必要性がないため、リポジトリ直下をモジュールルートにします。

#### 実行開始点

Web APIの実行開始点として、`cmd/api/main.go`を使用できます。

```text
cmd/
└── api/
    └── main.go
```

`cmd`は`command`の略で、実行可能プログラムの入口を置くために使われる慣習的なディレクトリ名です。

ただし、`main.go`に何を置き、どの処理を別のパッケージへ分けるかは、あらかじめ固定しません。

実装前後に、次を考えます。

- `main.go`は何を担当するべきか
- 起動処理とHTTP処理を分ける必要があるか
- DB処理はどこへ置くと変更・テストしやすいか
- 複数の処理で共通する責務が生じているか
- 新しいパッケージを作ることで何が改善されるか
- 分割によって依存関係が複雑になっていないか
- 現在のPhaseに対して過剰な構成ではないか

#### `internal/`の扱い

`internal/`は、このアプリ内部でのみ使用するパッケージを置く候補です。

ただし、最初から`handler/`、`service/`、`repository/`などの完成形を作りません。

必要な責務が実際に生じた時点で、次を比較して判断します。

```text
同じパッケージに置いたままにする
  または
新しいパッケージへ分ける
```

判断するときは、次の観点を使います。

- 責務を一文で説明できるか
- 変更理由が異なる処理を分けられるか
- テストしやすくなるか
- 依存関係が単純になるか
- 現在の規模で本当に必要か
- 将来の可能性だけを理由にしていないか

ディレクトリ名やレイヤー構成そのものを正解として写すのではなく、要求、責務、依存関係、テストのしやすさから自分で決めます。

#### 依存関係を考える

`cmd/api/main.go`から`internal/`の処理を利用する構成は一つの候補です。

```text
cmd/api
  ↓
internal
```

ただし、この形を採用する場合も、単に一般的だから選ぶのではなく、次を説明できることを重視します。

- なぜその依存方向にしたか
- 逆方向へ依存すると何が起きるか
- どの処理を再利用したいか
- どの部分をテスト時に置き換えたいか
- 別の実行方法を追加した場合に再利用できるか

#### 配置方針の確認項目

- [ ] `go.mod`を置く場所を説明できる
- [ ] `src/`を作る、または作らない理由を説明できる
- [ ] 実行開始点をどこへ置くか決めた
- [ ] `main.go`の責務を自分の言葉で説明できる
- [ ] 新しく追加したパッケージの責務を説明できる
- [ ] 将来使うかもしれないという理由だけで分割していない
- [ ] 現在のPhaseに対して過剰な構成になっていない
- [ ] 構成を変更した場合、その理由を設計メモへ記録した

### 共通の設計完了条件

各Phase固有の完了条件に加えて、次も確認します。

- [ ] 実装前に目的・制約・対象外を記録した
- [ ] 重要な箇所について複数案を比較した
- [ ] 採用した案の理由と欠点を説明できる
- [ ] 責務の置き場所と依存方向を説明できる
- [ ] 正常時だけでなく、失敗時の流れを考えた
- [ ] セキュリティ上、信用しない入力や境界を特定した
- [ ] 設計を検証するテスト方針を決めた
- [ ] 実装後に、当初の設計判断が妥当だったか振り返った
- [ ] 設計を変更した場合、その理由を設計メモまたはADRへ追記した
