# 設計方針

### 2.4 設計も実装と同じく反復して学ぶ

設計力は、最初から正解の構造を当てる能力ではありません。

このプロジェクトでは、設計を次のように扱います。

```text
要求から設計上の問題を見つける
  ↓
少なくとも2つの案を考える
  ↓
制約とトレードオフを比較する
  ↓
現時点の採用案を決める
  ↓
実装とテストで妥当性を確かめる
  ↓
必要であれば設計を変更する
```

設計資料には、完成したコードの実装手順やアルゴリズムを先に書きません。

記録するのは、主に次の内容です。

- 何を実現する必要があるか
- 何を今回の対象外にするか
- どの責務をどこへ置くか
- どのデータを誰が所有するか
- どのような失敗が起こり得るか
- どの境界を信用しないか
- どの設計案を比較したか
- なぜ現在の案を採用したか
- どのテストで設計の妥当性を確認するか

これにより、実装方法そのものは自分で考えながら、設計判断を言語化する練習ができます。

### 設計課題の進め方

各Phaseでは、実装前に短い設計メモを作成します。長大な設計書ではなく、実装前後で判断を比較できる程度の内容にします。

#### 設計メモに記載する内容

```text
目的
今回の対象範囲
対象外にすること
ユースケース
APIやデータの契約
主な責務と境界
依存関係
想定する失敗
セキュリティ上の信用境界
検討した設計案
採用案と理由
欠点・トレードオフ
テスト方針
未解決事項
```

すべての項目を毎回詳しく書く必要はありません。そのPhaseで重要な項目を選んで記載します。

#### ADRを残す判断

後から変更理由を知りたくなる重要な判断は、Architecture Decision Recordとして残します。

例：

- セッション方式をどのようにするか
- 認可をどの層で行うか
- Repository層を導入するか
- `403`と`404`をどう使い分けるか
- ファイルをどこへ保存するか
- 同時更新をどの方式で制御するか

ADRには、少なくとも次を記載します。

```text
背景
決める必要があること
検討した選択肢
採用した選択肢
採用理由
想定される影響
後から見直す条件
```

### Phaseごとの主な設計テーマ

| Phase | 設計テーマ |
|---|---|
| Phase 0 | クライアントとサーバーの境界、HTTPメッセージの責務 |
| Phase 1 | API契約、エラー形式、共通処理と個別処理の境界 |
| Phase 2 | リソース設計、URL設計、`PUT`と`PATCH`、バリデーション方針 |
| Phase 3 | 横断的関心事、ミドルウェアの責務と実行順序 |
| Phase 4 | 永続化境界、Repositoryの責務、トランザクション境界 |
| Phase 5 | 認証フロー、セッションのライフサイクル、秘密情報の管理 |
| Phase 6 | 認可ポリシー、データ所有権、存在情報を隠すかどうか |
| Phase 7 | ロール設計、管理者権限の不変条件、監査ログ |
| Phase 8 | ファイルの信用境界、保存責務、検証と配信の分離 |
| Phase 9 | 脅威モデル、攻撃経路、手動検査の優先順位 |
| Phase 10 | テスト階層、テストデータ生成、再現性と独立性 |
| Phase 11 | フロントエンドとAPIの責務分担、画面状態と通信エラー |
| Phase 12 | オリジン境界、ブラウザの信頼モデル、CSRF・XSS対策 |
| Phase 13 | 整合性、競合制御、タイムアウト、キャンセル伝播 |
| Phase 14 | 実行環境、設定管理、可観測性、デプロイと運用 |

### Goソースコードの配置方針

このプロジェクトでは、アプリのソースコードを`src/`配下へまとめず、リポジトリ直下に`go.mod`を置き、必要に応じて`cmd/`と`internal/`へ分けます。

基本構成は次のとおりです。

```text
go-http-security-lab/
├── go.mod
├── go.sum
├── cmd/
│   ├── api/
│   │   └── main.go
│   └── doccheck/
│       └── main.go
├── internal/
│   ├── handler/
│   ├── middleware/
│   ├── model/
│   ├── repository/
│   ├── service/
│   ├── session/
│   ├── validation/
│   └── response/
├── migrations/
├── tests/
├── web/
├── api-client/
├── docs/
└── README.md
```

`cmd/api/`にはWeb APIの実行開始点を置きます。

将来、ドキュメント検査CLIなど別の実行プログラムを追加する場合は、`cmd/doccheck/`のように別のディレクトリを作ります。

`internal/`には、このアプリの内部でのみ使用するパッケージを置きます。

#### `src/`を作らない理由

Goでは、`go.mod`が置かれた場所がモジュールの基準になります。

リポジトリ直下に`go.mod`を置いた状態で`src/`を追加すると、`src`がそのままパッケージパスの一部になります。

```text
github.com/example/go-http-security-lab/src/internal/handler
```

この`src`には、今回のGoアプリにおける特別な意味がありません。

単一のGoモジュールとしては、次のように直接配置した方が構造を理解しやすくなります。

```text
github.com/example/go-http-security-lab/internal/handler
```

また、`src/`の中へ`go.mod`を置く構成も可能ですが、リポジトリルートとGoモジュールルートが分かれます。

```text
go-http-security-lab/
├── src/
│   ├── go.mod
│   ├── cmd/
│   └── internal/
├── docs/
└── README.md
```

この構成では、リポジトリルートからそのまま`go test ./...`などを実行しにくくなります。

今回のプロジェクトは、一つのGoアプリを中心とする単一モジュールなので、リポジトリ直下をモジュールルートにします。

#### 最初から完成形の構成を作らない

最終的な推奨構成を、Phase 1の開始時点からすべて作る必要はありません。

Phase 1では、次のような最小構成から始めても構いません。

```text
go-http-security-lab/
├── go.mod
├── main.go
└── main_test.go
```

HTTP処理が増え、役割を分ける必要が生じたら、実行開始点と内部処理を分離します。

```text
go-http-security-lab/
├── go.mod
├── cmd/
│   └── api/
│       └── main.go
└── internal/
    ├── handler/
    └── response/
```

データベース、認証、認可などが加わった段階で、必要な責務だけを追加します。

```text
internal/
├── handler/
├── middleware/
├── repository/
├── service/
└── session/
```

この進め方により、ディレクトリ構成を先に写すのではなく、次のことを経験できます。

- なぜ処理を分ける必要が生じたか
- どの責務をどこへ置くべきか
- 分割によってテストしやすくなったか
- 依存関係がどの方向を向いているか
- 分割が現在の規模に対して過剰ではないか

#### ディレクトリを追加する判断基準

新しいパッケージやディレクトリは、名前を付けられる具体的な責務が生じたときに追加します。

追加を検討する例：

```text
HTTP固有の処理を分離したい
DBアクセスを交換・テストしたい
複数のHandlerで共通する業務ルールが生じた
認証状態やセッション管理を独立させたい
入力検証が複数箇所へ広がった
レスポンス形式を統一したい
```

追加しない方がよい例：

```text
有名なプロジェクトに同じディレクトリがある
将来使うかもしれない
ファイルが少し長くなった
レイヤー数を増やした方が高度に見える
```

#### `service/`を必須にしない

`handler/`、`repository/`、`service/`という構成を、最初から固定しません。

独立した業務ルールがほとんどない段階では、`service/`が単なる処理の受け渡しになる可能性があります。

```text
Handler
  ↓
Service
  ↓
Repository
```

この形を採用する場合は、Serviceがどのような責務を持つのか説明できることを条件にします。

例：

- 複数のRepositoryを組み合わせる
- トランザクションの単位を決める
- HTTPに依存しない業務ルールを実行する
- 認可を含むユースケースをまとめる
- 状態遷移の条件を管理する

責務を説明できない場合は、無理に作らず、実際に必要になった時点で追加します。

#### 配置方針の完了条件

- [ ] `go.mod`がリポジトリ直下にある
- [ ] 特別な理由がない限り`src/`を作っていない
- [ ] 実行開始点とアプリ内部の処理を区別できる
- [ ] 新しく追加したディレクトリの責務を説明できる
- [ ] 空のディレクトリや将来用のレイヤーを先に作っていない
- [ ] パッケージ分割によって依存関係が複雑になっていない
- [ ] 現在のPhaseに対して過剰な構成になっていない
- [ ] 構成を変更した理由を設計メモまたはADRへ記録した

### 共通の設計完了条件

各Phase固有の完了条件に加えて、次も確認します。

- [ ] 実装前に目的・制約・対象外を記録した
- [ ] 重要な箇所について複数案を比較した
- [ ] 採用した案の理由と欠点を説明できる
- [ ] 責務の置き場所と依存方向を説明できる
- [ ] 正常時だけでなく、失敗時の流れを考えた
- [ ] セキュリティ上、信用しない入力や境界を特定した
- [ ] 設計を検証するテスト方針を決めた
- [ ] 実装後に、当初の設計判断が妥当だったか振り返った
- [ ] 設計を変更した場合、その理由を設計メモまたはADRへ追記した
