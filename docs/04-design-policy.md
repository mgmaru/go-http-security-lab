# 設計方針

### 2.4 設計も実装と同じく反復して学ぶ

設計力は、最初から正解の構造を当てる能力ではありません。

このプロジェクトでは、設計を次のように扱います。

```text
要求から設計上の問題を見つける
  ↓
少なくとも2つの案を考える
  ↓
制約とトレードオフを比較する
  ↓
現時点の採用案を決める
  ↓
実装とテストで妥当性を確かめる
  ↓
必要であれば設計を変更する
```

設計資料には、完成したコードの実装手順やアルゴリズムを先に書きません。

記録するのは、主に次の内容です。

- 何を実現する必要があるか
- 何を今回の対象外にするか
- どの責務をどこへ置くか
- どのデータを誰が所有するか
- どのような失敗が起こり得るか
- どの境界を信用しないか
- どの設計案を比較したか
- なぜ現在の案を採用したか
- どのテストで設計の妥当性を確認するか

これにより、実装方法そのものは自分で考えながら、設計判断を言語化する練習ができます。

### 設計課題の進め方

各Phaseでは、実装前に短い設計メモを作成します。長大な設計書ではなく、実装前後で判断を比較できる程度の内容にします。

#### 設計メモに記載する内容

```text
目的
今回の対象範囲
対象外にすること
ユースケース
APIやデータの契約
主な責務と境界
依存関係
想定する失敗
セキュリティ上の信用境界
検討した設計案
採用案と理由
欠点・トレードオフ
テスト方針
未解決事項
```

すべての項目を毎回詳しく書く必要はありません。そのPhaseで重要な項目を選んで記載します。

#### ADRを残す判断

後から変更理由を知りたくなる重要な判断は、Architecture Decision Recordとして残します。

例：

- セッション方式をどのようにするか
- 認可をどの層で行うか
- Repository層を導入するか
- `403`と`404`をどう使い分けるか
- ファイルをどこへ保存するか
- 同時更新をどの方式で制御するか

ADRには、少なくとも次を記載します。

```text
背景
決める必要があること
検討した選択肢
採用した選択肢
採用理由
想定される影響
後から見直す条件
```

### Phaseごとの主な設計テーマ

| Phase | 設計テーマ |
|---|---|
| Phase 0 | クライアントとサーバーの境界、HTTPメッセージの責務 |
| Phase 1 | API契約、エラー形式、共通処理と個別処理の境界 |
| Phase 2 | リソース設計、URL設計、`PUT`と`PATCH`、バリデーション方針 |
| Phase 3 | 横断的関心事、ミドルウェアの責務と実行順序 |
| Phase 4 | 永続化境界、Repositoryの責務、トランザクション境界 |
| Phase 5 | 認証フロー、セッションのライフサイクル、秘密情報の管理 |
| Phase 6 | 認可ポリシー、データ所有権、存在情報を隠すかどうか |
| Phase 7 | ロール設計、管理者権限の不変条件、監査ログ |
| Phase 8 | ファイルの信用境界、保存責務、検証と配信の分離 |
| Phase 9 | 脅威モデル、攻撃経路、手動検査の優先順位 |
| Phase 10 | テスト階層、テストデータ生成、再現性と独立性 |
| Phase 11 | フロントエンドとAPIの責務分担、画面状態と通信エラー |
| Phase 12 | オリジン境界、ブラウザの信頼モデル、CSRF・XSS対策 |
| Phase 13 | 整合性、競合制御、タイムアウト、キャンセル伝播 |
| Phase 14 | 実行環境、設定管理、可観測性、デプロイと運用 |

### 共通の設計完了条件

各Phase固有の完了条件に加えて、次も確認します。

- [ ] 実装前に目的・制約・対象外を記録した
- [ ] 重要な箇所について複数案を比較した
- [ ] 採用した案の理由と欠点を説明できる
- [ ] 責務の置き場所と依存方向を説明できる
- [ ] 正常時だけでなく、失敗時の流れを考えた
- [ ] セキュリティ上、信用しない入力や境界を特定した
- [ ] 設計を検証するテスト方針を決めた
- [ ] 実装後に、当初の設計判断が妥当だったか振り返った
- [ ] 設計を変更した場合、その理由を設計メモまたはADRへ追記した
