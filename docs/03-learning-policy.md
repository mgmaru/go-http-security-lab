# 学習方針

### 2.1 コーディングを中心に学ぶ

このプロジェクトでは、教材を読む時間よりも、実装・検証・修正に多くの時間を使います。

目安として、学習時間を次のように配分します。

```text
実装・リファクタリング：50%
テストコード作成：20%
curl・Burp Suiteによる検証：20%
ドキュメント・教材確認：10%
```

教材をすべて読み終えてから実装するのではなく、必要になった知識をその都度調べます。

### 2.2 最初はフロントエンドを作らない

前半は、次の3つをクライアントとして使います。

| ツール | 主な目的 |
|---|---|
| `curl` | HTTPリクエストを明示的に送る |
| Go `httptest` | APIの挙動を自動検証する |
| Burp Suite | 通信を取得・変更・再送する |

フロントエンドを最初から作ると、HTML、CSS、画面設計、状態管理などに学習が分散します。

そのため、認証・認可までAPIを完成させた後で、ブラウザ固有の仕組みを学ぶために最小限のフロントエンドを追加します。

### 2.5 過剰設計を避ける

将来必要になるかもしれない機能を、根拠なく先回りして実装しません。

各Phaseでは、その時点の要求を満たす最小限の設計から始めます。後のPhaseで要求が増えたときに、どこが変更しにくかったかを観察し、必要な範囲だけリファクタリングします。

```text
単純な設計で開始する
  ↓
新しい要求を追加する
  ↓
変更しにくい箇所を発見する
  ↓
責務や依存関係を見直す
```

この「変更によって設計の弱点を発見する経験」も、設計学習の一部とします。

### 2.6 学習の過程を振り返りとして残す

このプロジェクトでは、完成したコードだけでなく、実装中に起きたことや考えた過程も成果物として残します。

記録する対象は次のとおりです。

- 実際に起きたこと
- 想定どおりだったこと
- 想定と異なったこと
- 発生したバグ
- テストやBurp Suiteで見つかった問題
- 新しく理解できたこと
- まだ理解できていないこと
- 判断に迷ったこと
- 採用しなかった選択肢
- 実装前の設計から変更したこと
- 次のPhaseへ持ち越すこと

この記録は単なる日記ではなく、次の目的で使用します。

```text
過去の問題と解決方法を再利用する
自分が迷いやすい判断を把握する
設計前の予想と実装後の結果を比較する
同じ疑問を繰り返し調べることを防ぐ
面接やポートフォリオで思考過程を説明する
```

#### 文書の役割分担

| 文書 | 主に記録する内容 |
|---|---|
| README | 学習計画、進め方、完了条件 |
| 設計メモ | 実装前の要求、制約、責務、設計案 |
| ADR | 将来にも影響する重要な設計判断 |
| Retrospective | 実装後に起きたこと、学び、迷い、反省 |
| 未解決事項 | まだ解決できていない疑問や調査項目 |

同じ内容をすべての文書へ重複して書く必要はありません。

```text
一時的な迷いや実装中の出来事
→ Retrospective

長期的に影響する設計判断
→ ADR

まだ結論が出ていない疑問
→ 未解決事項

今後の学習手順やルール
→ README
```

#### 記録するタイミング

##### 作業中

作業を止めないように、短いメモだけを残します。

```text
PATCHで未指定とゼロ値をどう区別するか迷った
壊れたJSONの扱いが想定と異なった
HandlerがDBエラーの詳細を知りすぎている気がする
```

##### Phase終了時

作業中のメモを整理して、Phaseごとの振り返りを作成します。

##### 重要な判断が発生したとき

将来も影響する判断だけをADRへ移します。

##### 疑問を解決できなかったとき

未解決事項の一覧へ登録し、忘れないようにします。

#### Retrospectiveに記載する項目

Phaseごとの振り返りには、必要に応じて次を記載します。

```text
このPhaseで取り組んだこと
実装前に予想していたこと
実際に起きたこと
学んだこと
分からなかったこと
迷ったこと
失敗したこと
設計の振り返り
コードの振り返り
次のPhaseへ持ち越すこと
今回の成果
このPhaseで最も重要だった学び
```

毎回すべての項目を長く書く必要はありません。特に重要だった内容を中心に記載します。

#### 未解決事項の管理

解決していない疑問にはIDと状態を付けます。

```text
Q-001：PUTとPATCHをどの程度厳密に分けるか
発生Phase：Phase 2
状態：調査中
現在の理解：
まだ不明な部分：
次に調べる資料：
最終的な結論：
関連するADR：
```

状態の例：

```text
未調査
調査中
一旦保留
解決済み
設計判断済み
```

#### 振り返りの完了条件

各Phaseの完了時に、次を確認します。

- [ ] 実装中に起きた重要な出来事を記録した
- [ ] 新しく学んだことを自分の言葉で説明した
- [ ] 分からなかったことを隠さず記録した
- [ ] 迷った判断と比較した選択肢を記録した
- [ ] 実装前の設計から変わった点を記録した
- [ ] 未解決の疑問を未解決事項へ登録した
- [ ] 次のPhaseへ持ち越す課題を明確にした
- [ ] 重要な設計判断を必要に応じてADRへ移した

### 2.7 取り組み上の注意事項

このプロジェクトは、実装、設計、テスト、セキュリティ検証、振り返りまで扱います。

扱う範囲が広いため、次の注意事項を守って進めます。

#### 完璧になるまで同じPhaseへ留まらない

各Phaseでは、その時点での最善を目指しますが、完全無欠な設計やコードを求め続けません。

次の条件を満たしたら、残っている改善点を記録して次のPhaseへ進んで構いません。

```text
そのPhaseの中心的な機能が動作する
主要な正常系・異常系テストが成功する
重要なセキュリティ上の問題を放置していない
未解決事項と改善候補を記録している
自分の設計判断を説明できる
```

後のPhaseで新しい要求が加わり、以前の設計の弱点が見つかることも学習の一部です。

#### コードを書く時間を最優先にする

設計メモ、ADR、Retrospectiveは重要ですが、文書作成そのものを目的にしません。

学習時間の目安は次のとおりです。

```text
実装・リファクタリング・テスト：70〜80%
調査・curl・Burp Suiteによる検証：10〜20%
設計メモ・ADR・振り返り：10〜15%
```

割合は厳密なものではありません。

文書が実装時間を圧迫している場合は、まず短い箇条書きで記録し、Phase終了時に必要な部分だけ整理します。

#### 答えをすぐに写さない

実装力を高めるため、実装方法を調べる前に、まず自分で次を行います。

```text
要求を分解する
入力と出力を整理する
処理の流れを考える
候補となる設計を複数考える
小さな単位で試作する
テストケースを先に考える
```

分からない場合は、完成コードをそのまま取得するのではなく、次の順番で助けを求めます。

```text
公式ドキュメントを読む
概念や仕様を調べる
小さな実験コードを書く
ヒントやレビューを求める
最後に必要な範囲だけ実装例を確認する
```

参考コードを見た場合でも、コピーして終わらせず、各行の役割と別の実装案を説明できる状態にします。

#### 抽象化を先回りしない

将来使うかもしれないという理由だけで、インターフェース、共通化、汎用パッケージを増やしません。

次のような具体的な問題が起きてから、抽象化を検討します。

- 同じ処理が複数箇所で繰り返されている
- 変更のたびに多くの場所を修正する必要がある
- テストで依存先を置き換えられない
- 異なる責務が一つの関数や型に混在している
- 実装詳細が上位層へ漏れている

単にコードが長くなったことだけを理由に分割せず、変更理由と責務の違いを基準にします。

#### 動作する状態を小さく保つ

一度に大量の機能を実装せず、小さな単位で動作を確認します。

```text
一つの小さな機能を実装する
  ↓
テストする
  ↓
curlで確認する
  ↓
コミットする
  ↓
次の機能へ進む
```

大きな変更を始める前に、現在のテストが成功することを確認します。

意図的に脆弱な状態を作る場合は、ブランチやタグで安全な状態と区別します。

#### テストが通ることを安全性の証明と考えない

自動テストや脆弱性スキャンが成功しても、すべての問題がなくなったことにはなりません。

特に次の問題は、アプリ固有の仕様を理解しなければ発見できない場合があります。

- 他人のデータへアクセスできる
- 一般ユーザーが管理者操作を実行できる
- 状態遷移の順序を変えると不正な操作ができる
- 同じリクエストの再送でデータが重複する
- 複数処理の途中失敗でデータが不整合になる

自動テスト、コードレビュー、Burp Suiteによる手動検査を組み合わせます。

#### セキュリティ検証の対象を限定する

このリポジトリには、意図的または偶発的な脆弱性が含まれる可能性があります。

必ず次を守ります。

- localhostまたは自分が管理する隔離環境で実行する
- 脆弱な状態のアプリをインターネットへ公開しない
- Burp Suiteは自分が所有するか、明示的な許可を得た対象だけに使用する
- 実在する個人情報を使用しない
- 本物のパスワードを使用しない
- 本物のAPIキーやアクセストークンを使用しない
- 学習用の脆弱なコードを本番環境へ流用しない
- アップロードしたファイルを不用意に実行しない

#### 秘密情報をGitへ入れない

次の情報はコミットしません。

```text
.env
実際のパスワード
セッションID
秘密鍵
APIキー
アクセストークン
個人情報を含むDBファイル
本物のアップロード画像
```

誤ってコミットした場合は、ファイルを削除するだけでなく、秘密情報を失効・再発行し、Git履歴への対応も行います。

#### 行き詰まりを記録して区切る

一つの疑問へ長時間取り組み続けて進行が止まった場合は、次のように区切ります。

```text
何が分からないかを文章にする
試したことを記録する
現在の仮説を記録する
未解決事項へ登録する
Phaseの完了に必須か判断する
```

完了に必須でなければ一旦保留し、後で調査します。

必須であれば、問題をさらに小さく分けて実験します。

#### READMEは必要に応じて更新する

実際に進めた結果、期間、順序、完了条件、ライブラリ方針が現実に合わない場合は変更して構いません。

READMEを絶対的な仕様として扱うのではなく、学習経験に応じて改善する計画書として扱います。

### 2.8 想定期間と進行コース

このプロジェクトは実装量が多いため、全Phaseを短期間で完了させることは想定していません。

期間は、理解を飛ばして期限に合わせるためのノルマではなく、次の目的で使用します。

- 一つのPhaseへ留まり続けていないか確認する
- 実際の作業量を把握する
- 自分が時間を使いやすい分野を把握する
- 学習計画を現実に合わせて調整する
- 途中で学習範囲を区切る判断材料にする

#### 想定する学習時間

標準的な目安は次のとおりです。

```text
週10〜15時間
全Phase：18〜24週間程度
全体：約4〜6か月
```

GoやWeb APIの経験、テスト量、ドキュメント量、追加課題によって期間は変わります。

特にPhase 4〜6、Phase 12〜13は、設計と検証に時間がかかる可能性があります。

#### 進行コース

学習目的や使える期間に応じて、途中を一つの到達点として扱います。

| コース | 対象 | 期間の目安 | 主に身につける内容 |
|---|---:|---:|---|
| 基礎コース | Phase 0〜3 | 3〜6週間 | HTTP、Go API、CRUD、ミドルウェア |
| バックエンド・セキュリティコース | Phase 0〜10 | 3〜4か月 | DB、認証、認可、ファイル、Burp Suite、セキュリティテスト |
| 完全コース | Phase 0〜14 | 4〜6か月以上 | ブラウザセキュリティ、同時実行、CI、運用を含む全体 |

Phase 0〜10まででも、GoバックエンドとWebセキュリティの学習プロジェクトとして十分な成果になります。

#### Phase別の期間目安

| Phase | 内容 | 期間の目安 |
|---:|---|---:|
| 0 | 開発環境とHTTP通信の観察 | 1〜3日 |
| 1 | 基本HTTPサーバー | 1週間 |
| 2 | タスクCRUD API | 1〜2週間 |
| 3 | ルーターとミドルウェア | 1週間 |
| 4 | SQLiteとRepository層 | 2〜3週間 |
| 5 | ユーザー登録・ログイン・セッション | 2〜3週間 |
| 6 | ユーザーごとの認可 | 1〜2週間 |
| 7 | 管理者機能 | 1週間 |
| 8 | ファイルアップロード | 1〜2週間 |
| 9 | Burp Suiteによる手動検査 | 1週間 |
| 10 | セキュリティテスト化 | 1〜2週間 |
| 11 | 最小限のフロントエンド | 1〜2週間 |
| 12 | CORS・CSRF・XSS | 2〜3週間 |
| 13 | 同時実行・競合・タイムアウト | 1〜2週間 |
| 14 | CI・静的解析・仕上げ | 1〜2週間 |

期間内に終わらない場合でも、理解を飛ばして次へ進む必要はありません。

一方で、完璧なリファクタリングだけを続けている場合は、未解決事項として記録し、次のPhaseへ進むことを検討します。

#### 1週間の進め方の例

週10〜15時間の場合の一例です。

```text
1日目：要求整理・設計メモ・テストケース検討
2〜4日目：実装・リファクタリング
5日目：httptest・統合テスト
6日目：curl・Burp Suite・異常系の検証
7日目：振り返り・未解決事項・次週の準備
```

毎日取り組む必要はありません。実際の生活に合わせて、同じ内容を複数日に分けても構いません。

#### Phaseの進捗記録

Phaseごとに、予定と実績を記録します。

```markdown
## Phase XX 進捗

- 開始日：
- 完了予定日：
- 完了日：
- 予定期間：
- 実際の経過期間：
- 実際の作業時間：
- 現在の状態：未着手 / 進行中 / 保留 / 完了
- 予定との差：
- 差が生じた理由：
- 主な障害：
- 次の調整：
```

期間だけでなく、可能であれば実際の作業時間も記録します。

経過日数が長くても、実作業時間が少ない場合があります。両方を記録することで、課題の難易度と生活上の都合を分けて振り返れます。

#### 期間を見直すタイミング

次のタイミングで、以降の予定を調整します。

- Phase終了時
- 4週間ごと
- 予定の2倍以上かかったとき
- 生活や学習時間が大きく変化したとき
- 学習目的や転職計画が変化したとき
- 追加課題を大幅に増やしたとき

予定との差は失敗ではありません。

差が生じた理由を分析し、次の見積もりへ反映すること自体が、開発における見積もりと計画調整の練習になります。

## 方針の見直し

実際の進捗に合わなくなった場合は、理由を記録したうえで期間、順序、完了条件を更新します。
