# 標準ライブラリと外部ライブラリの方針

### 2.3 Goの標準ライブラリと外部ライブラリを使い分ける

このプロジェクトでは、すべてを標準ライブラリだけで実装することを目的にはしません。

基本方針は次のとおりです。

```text
今回の学習対象を理解するための処理
→ できるだけ自分で実装する

暗号・認証など、安全性が重要な基盤処理
→ 信頼できる既存ライブラリを使う

データベース接続など、標準ライブラリだけでは完結しない処理
→ 必要な外部ライブラリを使う

便利さだけを目的に学習対象を隠すライブラリ
→ 基礎を実装するまでは導入しない
```

外部ライブラリを使うこと自体を避けるのではなく、**何を自分で考えて実装し、何を既存の仕組みに任せるかを判断すること**も設計学習の一部とします。

#### 自分で実装するもの

このプロジェクトの中心的な学習対象は、自分で設計・実装します。

- HTTPハンドラー
- APIのルーティング方針
- リクエストとレスポンスの変換
- エラーレスポンス
- 入力値検証
- ミドルウェアの構成
- 認証後のアプリケーション処理
- 認可ルール
- タスクの所有権
- RepositoryとServiceの責務分担
- トランザクション境界
- セキュリティテスト
- 同時更新時のアプリケーション上の扱い

#### 独自実装しないもの

安全性が重要で、独自実装による危険が大きいものは、信頼できる実装を利用します。

- パスワードハッシュのアルゴリズム
- 暗号化アルゴリズム
- 暗号学的に安全な乱数生成
- TLSそのもの
- 電子署名のアルゴリズム
- セキュリティトークンの暗号学的処理

ただし、既存ライブラリを使う場合でも、次は自分で設計します。

```text
いつハッシュ化するか
何を保存するか
セッションをいつ作成・失効するか
どのCookie属性を設定するか
認証失敗をどのように扱うか
誰にどの操作を許可するか
```

#### 外部ライブラリを導入する判断基準

外部ライブラリを追加する前に、次を確認します。

- [ ] 標準ライブラリでは実現できない、または安全に実現しにくいか
- [ ] 今回の学習対象を隠してしまわないか
- [ ] 単にコードを短くするためだけの導入ではないか
- [ ] メンテナンス状況や利用実績を確認したか
- [ ] 依存関係が過度に増えないか
- [ ] そのライブラリが停止・変更された場合の影響を説明できるか
- [ ] ライブラリ固有の型や処理がアプリ全体へ漏れすぎないか
- [ ] テスト時に置き換えやすい境界を作れるか
- [ ] なぜ導入したかを設計メモまたはADRへ記録したか

#### Phaseごとの方針

| Phase | ライブラリ方針 | 主な理由 |
|---|---|---|
| Phase 0 | 外部ライブラリ不要 | ブラウザ開発者ツールと`curl`でHTTPを観察する |
| Phase 1 | Go標準ライブラリのみ | `net/http`、JSON、メソッド、ヘッダー、エラー処理を直接理解する |
| Phase 2 | 原則としてGo標準ライブラリのみ | CRUD、URL、クエリ、バリデーションを自分で設計する |
| Phase 3 | 原則としてGo標準ライブラリのみ | ミドルウェア、ログ、Recovery、依存関係を直接実装する |
| Phase 4 | SQLiteドライバーなど必要最小限を許可 | `database/sql`だけではDBへ接続できないため、接続用ドライバーは利用する |
| Phase 5 | セキュリティ基盤のライブラリを利用 | パスワードハッシュなどの暗号学的処理は独自実装しない |
| Phase 6 | 原則として追加ライブラリ不要 | 認可と所有権はアプリ固有のルールなので自分で設計する |
| Phase 7 | 原則として追加ライブラリ不要 | ロール、管理者制約、監査方針を自分で設計する |
| Phase 8 | 標準ライブラリ中心。保存先に応じて追加を許可 | multipart、サイズ制限、検証は自分で扱い、クラウド保存時のみSDKなどを利用する |
| Phase 9 | Burp Suiteを使用 | 手動セキュリティ検査のための外部ツールとして利用する |
| Phase 10 | `httptest`中心。補助ライブラリは任意 | まず標準機能でテスト構造を理解し、重複が明確になった後で補助を検討する |
| Phase 11 | 最初は素のHTML・JavaScriptを推奨 | `fetch`、Cookie、CORSなどブラウザのHTTP動作を直接理解する |
| Phase 12 | 標準機能中心。必要時のみ安全なサニタイザーを利用 | 出力エスケープやCSPを理解し、任意HTMLを許可する場合は独自サニタイズを避ける |
| Phase 13 | Go標準ライブラリとDB機能を中心にする | goroutine、`context`、Race Detector、競合制御を直接学ぶ |
| Phase 14 | CI・静的解析・脆弱性検査ツールを導入する | 実務的な品質管理と依存関係管理を自動化する |

#### Phase 4以降もWebフレームワークは必須ではない

Phase 4以降でデータベースや認証を追加しても、途中でGinやEchoなどのWebフレームワークへ移行する必要はありません。

このプロジェクトの本線では、最後まで`net/http`を使い続けても構いません。

フレームワークを試す場合は、本線を書き換える前に、別ブランチや別ディレクトリで比較します。

```text
標準ライブラリ版を完成させる
  ↓
同じAPIの一部をフレームワークで試す
  ↓
コード量・責務・テスト・依存関係を比較する
  ↓
採用するか判断する
```

比較時には、次を記録します。

- 何が簡潔になったか
- 何がフレームワークに隠れたか
- テストしやすさは変わったか
- エラー処理やミドルウェアの設計はどう変わったか
- フレームワーク固有の依存がどこまで広がったか
- この規模のアプリに本当に必要だったか

#### 補助ライブラリを導入するタイミング

テスト、バリデーション、ロギングなどの補助ライブラリは、最初から導入しません。

まず自分で一度実装し、次のような具体的な問題が発生してから検討します。

```text
同じテスト準備が大量に重複した
バリデーション規則の管理が難しくなった
ログ形式の一貫性を保てなくなった
手作業のマイグレーション管理で事故が起きそうになった
```

「将来便利そうだから」ではなく、**現在発生している問題を解決するために導入する**ことを原則とします。

#### 依存関係の記録

外部ライブラリを追加した場合は、PRまたは設計メモへ次を記録します。

```text
ライブラリ名
利用するPhase
解決したい問題
標準ライブラリだけで対応しなかった理由
比較した候補
採用理由
アプリ内で利用する範囲
想定するリスク
削除・交換するときの影響
```

これにより、`go.mod`へ依存関係を追加する行為も、設計判断として扱います。
