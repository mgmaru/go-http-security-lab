# AIの利用ルール

### 2.10 AIの利用ルール

このプロジェクトではAIを使用して構いません。

ただし、AIを最初の設計者・実装者として使うのではなく、**自分で考えて実装した後に参加するレビュアー、相談相手、反対意見を出す相手**として使用します。

基本方針は次のとおりです。

```text
自分
→ 最初の設計者・実装者・最終判断者

AI
→ レビュアー・質問相手・見落としを指摘する相手
```

AI利用の目的は、完成コードを早く得ることではありません。

次の能力を伸ばすために使用します。

- 自分の考えを言語化する
- 設計上の弱点を発見する
- 見落としていた選択肢を知る
- 反対意見と比較する
- 不明点を切り分ける
- 自分の判断理由をより明確にする

#### 原則となる順序

設計や実装についてAIへ相談する前に、次の順序で進めます。

```text
要求と制約を自分で整理する
  ↓
設計案を自分で考える
  ↓
比較した選択肢と迷いを記録する
  ↓
自分で実装する
  ↓
テストと手動確認を行う
  ↓
設計上の疑問を自分の言葉で書く
  ↓
AIへレビューを依頼する
  ↓
AIの提案を自分で評価する
  ↓
最終判断を自分の言葉で記録する
```

「自分の納得のいくまで」とは、完璧になるまで無制限に考え続けることではありません。

少なくとも次の状態になってから、設計・実装についてAIへ相談します。

- [ ] 自分なりの設計案がある
- [ ] なぜその案を考えたか説明できる
- [ ] 少なくとも一つは別の選択肢を考えた
- [ ] 動作する、または問題を再現できるコードがある
- [ ] 試したことと結果を記録している
- [ ] 何が疑問なのかを文章にできる

#### 最初からAIへ聞かないもの

次は、このプロジェクトで伸ばしたい中心的な能力です。

原則として、最初に自分で考えます。

- アプリ全体の設計
- API設計
- パッケージ構成
- 責務分割
- データモデル
- 認証・認可の方針
- 関数や型の分割
- アルゴリズム
- テストケースの洗い出し
- リファクタリング方針
- 完成コード
- Phase課題の解答

次のような依頼は、最初の段階では行いません。

```text
このアプリを最適な構成で設計してください
このPhaseのコードをすべて書いてください
この要件に対する正解のパッケージ構成を決めてください
必要なテストケースをすべて作ってください
```

#### 最初からAIへ聞いてよいもの

今回の中心的な学習対象ではない操作知識や、事実確認には早い段階からAIを利用できます。

- ツールの操作方法
- コマンドのオプション
- エラーメッセージの意味
- 用語の意味
- 公式ドキュメントの探し方
- 環境構築上の問題
- GoやHTTPの仕様確認
- 学習資料の探し方
- Bruno、Burp Suite、SQLite CLIなどの基本操作

例：

```text
curlでCookieを保存するオプションは何か
Brunoで環境変数を設定する方法は何か
Burp SuiteでリクエストをRepeaterへ送る方法は何か
go testで特定のテストだけを実行する方法は何か
405と404は仕様上どのように異なるか
```

ただし、ツールの操作方法を聞いた場合でも、ツールが裏で何をしているかは確認します。

```text
操作方法
→ AIへ聞いてよい

ツールが生成したHTTP通信や設定の意味
→ 自分で確認する
```

ツールは将来置き換わる可能性がありますが、HTTPプロキシ、APIコレクション、環境変数、Cookie、デバッグなどの基礎概念は残ります。

#### AIへ相談するときの依頼方法

最初に完成コードを求めず、次の順で相談の深さを上げます。

| レベル | AIへ求める内容 |
|---:|---|
| 1 | 用語、仕様、公式情報の確認 |
| 2 | 考えるための観点や確認項目 |
| 3 | 自分の設計・コードへのレビュー |
| 4 | 小さなヒント |
| 5 | 問題箇所に限定した部分例 |
| 6 | 完成例 |

原則として、レベル1〜3を中心に使用します。

レベル4以降へ進む前に、なぜ現在の情報だけでは進めないのかを記録します。

学習効果の高い質問例：

```text
私はHandlerとRepositoryに分けました。

HTTP処理とDB処理を分離することが理由です。
ただし、業務ルールが少ない現段階でServiceを追加すべきか迷っています。

次について指摘してください。

- 責務が曖昧な箇所
- 過剰設計になっている箇所
- 見落としている失敗ケース
- 反対の設計案

完成コードは示さないでください。
```

避ける質問例：

```text
最適な設計に直して、完成コードをすべて出してください。
```

#### AIへ期待する役割

AIへは、正解の断定よりも次を求めます。

- 問題点の指摘
- 反対意見
- 見落としている失敗ケース
- 過剰設計の可能性
- 責務や依存関係の曖昧さ
- セキュリティ上の前提
- テストしにくい箇所
- 別の設計案とトレードオフ
- 追加で自分が考えるべき問い

AIの回答も誤る可能性があります。

提案を受けた後は、次を自分で確認します。

```text
現在の要件に本当に必要か
存在しない要件をAIが補っていないか
今のPhaseに対して過剰ではないか
自分の案より単純になっているか
依存関係が増えすぎないか
テストしやすくなるか
公式仕様や一次資料と矛盾していないか
```

#### 実装例を見る場合の条件

AIから実装例を見ること自体は禁止しません。

ただし、次の条件を満たしてから、問題箇所に限定して確認します。

- [ ] 自分で十分に試した
- [ ] 何が分からないか説明できる
- [ ] 試した方法と結果を記録した
- [ ] 問題を小さく切り分けた
- [ ] 完成コードではなく、必要な範囲だけを依頼した
- [ ] 見た後に一度閉じ、自分で書き直す
- [ ] なぜ動作するか説明する
- [ ] 別の入力や異常系を自分で追加する

避ける流れ：

```text
完成コードをコピーする
  ↓
動作したことだけ確認する
  ↓
理解しないまま次へ進む
```

推奨する流れ：

```text
自分で実装する
  ↓
詰まった理由を言語化する
  ↓
必要な部分だけ例を見る
  ↓
例を閉じる
  ↓
自分で再実装する
  ↓
テストを追加する
  ↓
自分の実装との差を記録する
```

#### AI利用の記録

設計や実装についてAIへ相談した場合は、`docs/ai/consultations/`へ簡単な記録を残します。

```text
相談した内容
相談前の自分の考え
自分で試したこと
疑問に感じた点
AIから得た指摘
採用した提案
採用しなかった提案
最終的な判断
その判断をした理由
AIなしで説明・再実装できるか
関連する設計メモ・ADR・PR
```

すべての操作質問を記録する必要はありません。

次の相談を記録対象とします。

- 設計判断へ影響した相談
- コードの構造を変更した相談
- セキュリティ方針へ影響した相談
- 自分の認識が大きく変わった相談
- 実装例を確認した相談

#### AI利用の完了条件

各Phaseの終了時に、AIを使用した場合は次を確認します。

- [ ] AIへ相談する前の自分の案を残した
- [ ] 疑問を自分の言葉で記載した
- [ ] AIへ完成コードではなく、まずレビューや観点を求めた
- [ ] AIの回答をそのまま採用せず評価した
- [ ] 採用・不採用とその理由を記録した
- [ ] AIが生成したコードを使用した場合、その内容を説明できる
- [ ] AIなしで同じ方針を説明できる
- [ ] 必要に応じて設計メモ、ADR、Retrospectiveを更新した

このプロジェクトにおけるAI利用を一文で表すと、次のようになります。

> AIは自分の思考を始めるためではなく、自分で考えた内容を検証し、広げ、批評するために使う。
