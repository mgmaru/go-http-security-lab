# APIクライアントの利用方針

### 2.9 APIクライアントの利用方針

このプロジェクトでは、APIの実行に`curl`だけでなく、APIクライアントも使用します。

ただし、APIクライアントは`curl`の代替ではなく、HTTPを理解した後に、リクエストを整理・再利用するための道具として使用します。

基本的な役割分担は次のとおりです。

| ツール | 主な役割 |
|---|---|
| `curl` | HTTPメソッド、ヘッダー、Cookie、ボディを明示して理解する |
| Bruno | 正常系・異常系のリクエストを保存、整理、再利用する |
| Go `httptest` | APIの契約とセキュリティ要件を自動テストとして保証する |
| Burp Suite | 正しいリクエストを意図的に改変し、設計上の前提を検証する |
| ブラウザ開発者ツール | ブラウザが実際に送受信した通信を確認する |

学習の流れは次のようにします。

```text
新しいHTTP通信をcurlで作る
  ↓
送信内容とレスポンスを理解する
  ↓
Brunoへ保存して再利用できるようにする
  ↓
重要な契約をhttptestへ変換する
  ↓
Burp Suiteで仕様外の値へ改変する
```

#### 推奨するAPIクライアント

このプロジェクトでは、最初のAPIクライアントとして**Bruno**を推奨します。

主な理由は次のとおりです。

- APIリクエストをローカルファイルとして管理できる
- Gitで差分を確認しやすい
- リポジトリと一緒に正常系・異常系の利用例を残せる
- 環境変数や認証情報を環境ごとに分けられる
- GUIでの手動実行とCLIでの実行を組み合わせられる
- 個人学習でクラウドワークスペースを前提にしなくてよい

Postman、HTTPieなどを使っても構いませんが、複数のAPIクライアントを同時に本格運用しません。

まず一つのツールで、リクエストの保存・環境管理・実行・共有の考え方を学びます。

#### 導入するタイミング

Phase 0〜1では、原則として`curl`だけを使います。

HTTPの構造を理解する前にGUIへ依存すると、ツールが自動で追加したヘッダー、Cookie、リダイレクトなどを見落としやすいためです。

| Phase | APIクライアントの使い方 |
|---:|---|
| Phase 0〜1 | `curl`で通信の構造を理解する。Brunoはまだ使用しなくてよい |
| Phase 2 | CRUDの正常系・異常系リクエストをBrunoへ保存し始める |
| Phase 3〜4 | フォルダ分け、環境変数、共通ヘッダー、DBを使う一連の操作を整理する |
| Phase 5 | ユーザー登録、ログイン、ログアウト、Cookieの動きを保存する |
| Phase 6〜7 | ユーザーA・ユーザーB・管理者のリクエストを区別する |
| Phase 8 | `multipart/form-data`によるアップロード例を保存する |
| Phase 9 | Brunoに保存した正しいリクエストを基準としてBurp Suiteで改変する |
| Phase 10以降 | 重要なリクエストをGoの自動テストへ移し、Brunoは手動確認と利用例に使う |

#### Brunoで管理する内容

APIコレクションは、機能と目的で整理します。

```text
api-client/bruno/
├── environments/
├── health/
├── authentication/
├── tasks/
│   ├── normal/
│   └── invalid/
├── admin/
├── file-upload/
└── security-cases/
```

例：

```text
Tasks
├── List own tasks
├── Get own task
├── Get another user's task
├── Create valid task
├── Create task without title
├── Create task with unknown field
├── Update task
└── Delete task
```

正常系だけでなく、異常系や権限不足のリクエストも保存します。

ただし、Brunoのコレクションを自動テストの唯一の根拠にはしません。重要な仕様はGoのテストコードでも保証します。

#### APIクライアントを使うときのルール

1. 新しい種類のリクエストは、最初に`curl`で送る
2. メソッド、URL、ヘッダー、Cookie、ボディを説明できてからBrunoへ保存する
3. GUIが自動追加したヘッダーやCookieを確認する
4. 正常系だけでなく、異常系と権限不足の例も保存する
5. 環境変数と秘密情報を区別する
6. 本物のパスワード、セッションID、APIキーをGit管理しない
7. Brunoで確認した重要な挙動は`httptest`へ移す
8. セキュリティ検証はBrunoだけで終わらせず、Burp Suiteでも行う
9. コレクションが実際のAPI仕様と一致しているか、Phase終了時に確認する

#### 環境と秘密情報

Gitへ含めてよい設定と、含めてはいけない秘密情報を分けます。

Gitへ含めてもよい例：

```text
ローカルAPIのベースURL
テスト用の固定ユーザー名
秘密ではない環境名
リクエストのテンプレート
```

Gitへ含めてはいけない例：

```text
本物のパスワード
有効なセッションCookie
APIキー
アクセストークン
秘密鍵
個人情報
```

公開用の環境ファイルには、ダミー値またはプレースホルダーを記載します。

#### APIクライアントを使用しない方がよい状況

次の場合は、APIクライアントより`curl`や小さな実験コードを優先します。

- HTTPメッセージの構造を初めて学んでいる
- ツールが何を自動設定しているか分からない
- リダイレクトやCookieの送信を一段ずつ確認したい
- シェルスクリプトから最小限の再現手順を作りたい
- 問題の切り分けで、依存する機能を減らしたい

APIクライアントの操作を覚えること自体が目的になり、Goの実装やHTTPの理解を圧迫している場合は、一時的に使用を止めます。

#### APIクライアント利用の完了条件

Phase 2以降では、必要に応じて次を確認します。

- [ ] 新しいリクエストを最初に`curl`で確認した
- [ ] 保存したリクエストのHTTP要素を説明できる
- [ ] 正常系と主要な異常系をBrunoへ保存した
- [ ] 環境依存値を適切に分離した
- [ ] 秘密情報をGit管理していない
- [ ] コレクションの構成からAPIの主要な利用方法が分かる
- [ ] 重要な仕様をGoの自動テストでも保証している
- [ ] Burp Suiteで改変する基準となる正常なリクエストを用意した
